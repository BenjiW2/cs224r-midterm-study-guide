Start Here

The entire course is about one problem: an agent repeatedly sees a situation, picks an action, gets feedback, and tries to pick better actions next time. Most formulas are just different ways to answer "which action was better than expected?"

A Tiny Example

Imagine a robot in a hallway. Each second it can move left, move right, or stop. It gets +1 if it reaches the snack table and 0 otherwise.

RL word Hallway meaning What to picture
State s Where the robot is, maybe plus battery and obstacles The input
Action a Move left, move right, or stop The output
Policy π(a|s) Probabilities for actions in a state The model you train
Reward r +1 at the snack table, 0 otherwise The feedback
Return Total future reward The score for a rollout

Return: The Score Of A Whole Future

A reward is one piece of feedback at one time step. A return is the total reward you get from now into the future. In RL, the agent does not just ask "was this action good right now?" It asks "if I take this action, how good is the future I end up with?"

One-step reward: r_t = feedback at time t Return from time t: G_t = r_t + r_{t+1} + r_{t+2} + ... Discounted return: G_t = r_t + gamma r_{t+1} + gamma^2 r_{t+2} + ...

Undiscounted Example

The hallway robot gets rewards 0, 0, 1 because it reaches the snack table on the third step. The return at the beginning is 0 + 0 + 1 = 1.

Discounted Example

If gamma = 0.5, the same future has return 0 + 0.5 * 0 + 0.5^2 * 1 = 0.25. The reward still happens, but delayed reward counts less.

Why this matters: when policy gradient says "increase probability of actions in high-return trajectories," it means actions are credited according to the future total reward after them, not just the immediate reward at that same time step.

Three Questions Every Algorithm Asks

1. What data do I have?

Expert examples, fresh rollouts, a replay buffer, an offline dataset, human preferences, or a learned model.

2. How do I score an action?

Use raw return, a value function V, an action-value function Q, an advantage A, or a reward model.

3. How do I improve?

Copy good actions, increase their probability, choose the largest Q-value, or plan through a model.

How To Read The Symbols

πθ(a|s) Policy with parameters theta: probability of action a in state s "What would my neural net do here?"
Vπ(s) Value of state s if we follow policy π "How good is it to be here?"
Qπ(s,a) Value of taking action a in state s, then following π "How good is this action here?"

What is θ? Theta is not a new RL object. It just means "the parameters we are learning." If the policy is a neural network, θ is the giant vector containing every weight and bias in that network. So πθ(a|s) means: feed state s into the neural net with weights θ, and read off the probability it assigns to action a.

The One Mental Model To Keep

If action was better than expected: increase log πθ(a|s) If action was worse than expected: decrease log πθ(a|s) Advantage Aπ(s,a) = Qπ(s,a) - Vπ(s) is the "better than expected" number.
Sanity check: if A(s,a) > 0, should the policy make a more or less likely in s?
Answer. More likely. Positive advantage means the action was better than the policy's normal expected behavior in that state.

Math And ML Crash

Random Variables And Expectations

CS109 already gives you the main tool: an expectation is an average under a distribution. In RL, the random variable is often an entire trajectory, not a single label.

Discrete expectation: E[f(X)] = sum_x p(x) f(x) Trajectory expectation: E_{tau ~ p_theta(tau)}[R(tau)] means: average total reward over rollouts generated by policy pi_theta.

Gradients

A gradient is the answer to: "if I change the parameters a tiny bit, which direction makes the objective go up fastest?" If you have seen derivatives in Math 51, a gradient is just the multi-variable version of a derivative.

One Number Case

Suppose a policy has one parameter θ. If increasing θ makes reward go up, the derivative is positive, so gradient ascent increases θ. If increasing θ makes reward go down, the derivative is negative, so gradient ascent decreases θ.

Neural Net Case

A neural net has thousands or millions of parameters. The gradient has one entry per parameter. Each entry says whether that weight should move up or down, and by roughly how much.

Gradient ascent: used when maximizing reward θ ← θ + η ∇θ J(θ) Gradient descent: used when minimizing a loss θ ← θ - η ∇θ L(θ)

Here η is the learning rate: the step size. J(θ) usually means an objective we want to maximize, like expected return. L(θ) usually means a loss we want to minimize, like prediction error.

Worked Example: One Parameter

Suppose the policy has one parameter θ. Start at θ = 2. Use learning rate η = 0.1.

Gradient Ascent

We want to maximize reward:

J(θ) = 4θ - θ^2 Derivative: ∇θ J(θ) = 4 - 2θ At θ = 2: ∇θ J(2) = 4 - 2(2) = 0

The gradient is 0, so θ = 2 is already at the peak. No update is needed.

Now start at θ = 1 instead:

∇θ J(1) = 4 - 2(1) = 2 Ascent update: θ ← θ + η ∇θ J(θ) θ ← 1 + 0.1(2) = 1.2

Since the gradient is positive, increasing θ improves reward, so ascent moves θ upward from 1 to 1.2.

Gradient Descent

We want to minimize prediction error:

L(θ) = (θ - 3)^2 Derivative: ∇θ L(θ) = 2(θ - 3) At θ = 1: ∇θ L(1) = 2(1 - 3) = -4

Use descent, so subtract the gradient:

Descent update: θ ← θ - η ∇θ L(θ) θ ← 1 - 0.1(-4) θ ← 1.4

The gradient is negative. Subtracting a negative moves θ upward, toward the loss minimum at θ = 3.

Key sign rule: ascent adds the gradient because we want the objective higher. Descent subtracts the gradient because we want the loss lower.
RL translation: training changes θ, the neural-net weights, so that the policy assigns higher probability to actions that led to good future return and lower probability to actions that led to bad future return.

Maximum Likelihood

Maximum likelihood is the supervised-learning idea "make the data we actually saw have high probability under the model." In classification, if the correct label is "move right," we want the model to put high probability on "move right."

Action Policy probability before training Expert action? What training should do
left 0.20 No Usually push down
right 0.30 Yes Push up
stop 0.50 No Usually push down

The probability of the expert action is πθ(right|s) = 0.30. Behavior cloning updates θ so this number gets closer to 1 for expert examples.

min_theta E_{(s,a) ~ D_expert}[-log pi_theta(a | s)]

What This Equation Is Saying

Read it from the inside out. For each expert example, we look at the state s and the action a the expert actually took. Then we ask the policy: "What probability did you assign to that expert action?"

Piece Plain English
Dexpert The dataset of expert examples, like (state, expert action) pairs.
(s,a) ~ Dexpert Pick one expert example from the dataset.
πθ(a|s) The policy's probability of the expert's action a in state s.
-log πθ(a|s) The penalty. Small if the policy gave the expert action high probability; large if it did not.
𝔼 Average that penalty over all expert examples.
minθ Change the neural-net weights θ to make that average penalty as small as possible.

So the whole formula means: train the policy so that, across the expert dataset, the expert's action gets high probability.

Why -log? If the model gives the expert action high probability, -log(probability) is small. If it gives the expert action low probability, -log(probability) is large.

Probability assigned to expert action -log(probability) Interpretation
0.90 0.105 Small loss: model mostly agrees with expert
0.30 1.204 Medium loss
0.01 4.605 Large loss: model thought expert action was very unlikely

So behavior cloning is not mysterious RL yet. It is ordinary supervised learning: input is state s, label is expert action a, and the model is the policy πθ(a|s).

The Log-Derivative Trick

First, ignore the name. This trick is just the bridge from "I sampled an action and saw how well it worked" to "which way should I move the policy weights?"

In behavior cloning, an expert tells us the right action. In policy gradient RL, nobody tells us the right action. We try an action, see the return, and then use this rule:

Good return → make the sampled action more likely next time Bad return → make the sampled action less likely next time

Worked Example: One State, Two Actions

Suppose the robot is in one state and can choose right or left. Let p mean the policy's probability of choosing right.

Quantity Value
p(right) 0.30
p(left) 0.70

These are different possible rollouts, not one rollout where both actions happened. To make the numbers concrete, suppose "high return" means the robot eventually reaches the goal and gets G = +10.

Sampled action Return from that rollout What training should do Why
right G = +10 Increase p(right) right worked well, so make right more likely next time.
left G = +10 Decrease p(right) left worked well, so make left more likely. Since there are only two actions, this means lowering p(right).
If sampled action was right: log probability = log p derivative with respect to p = 1 / p at p = 0.30, this is positive with return G = +10, update pushes p upward If sampled action was left: log probability = log(1 - p) derivative with respect to p = -1 / (1 - p) at p = 0.30, this is negative with return G = +10, update pushes p downward

If the return were bad instead, the signs would flip. For example, if the policy sampled right and got G = -10, training would decrease p(right). If it sampled left and got G = -10, training would increase p(right), because making left less likely leaves more probability for right. In many algorithms, "good" and "bad" really mean above or below a baseline, not just positive or negative.

That is the whole idea of ∇ log probability: it points in the direction that would make the sampled action more likely. Then reward or return tells us whether we actually want to move in that direction.

Where The Equation Comes In

The policy-gradient objective is an average over possible rollouts. A rollout is random because the policy samples actions. If we write one possible sampled thing as x, its probability is pθ(x). The identity is:

grad_theta p_theta(x) = p_theta(x) grad_theta log p_theta(x)

It says: instead of directly differentiating the probability of the sampled thing, we can differentiate its log probability and multiply by its probability. This matters because "probability times something" is exactly an expectation, meaning we can estimate it by sampling.

Policy-gradient takeaway: sample a rollout compute how to make its sampled actions more likely: grad log probability multiply by how good it was: return update θ in that direction

Why Logs Specifically?

A rollout probability is a product of action probabilities:

p_theta(trajectory) = π_theta(a_1 | s_1) × π_theta(a_2 | s_2) × ... × π_theta(a_T | s_T)

Logs turn products into sums:

log p_theta(trajectory) = log π_theta(a_1 | s_1) + log π_theta(a_2 | s_2) + ... + log π_theta(a_T | s_T)

That is why the policy-gradient formula later has a sum of ∇ log πθ(at|st) terms.

Check: why do we use log probabilities instead of probabilities directly?
Answer. Products of probabilities appear in trajectory distributions. Logs turn products into sums, so log p_theta(tau) becomes a sum of log pi_theta(a_t | s_t). Also, grad p = p grad log p is what makes the policy-gradient estimator sampleable.

MDPs And Rewards

The RL Setup

An MDP, or Markov decision process, formalizes sequential decision-making.

Symbol Meaning Plain English
s_t state the information that fully describes the world for predicting the future
o_t observation what the agent actually sees; may be partial
a_t action the decision made by the agent
p(s_{t+1} | s_t, a_t) dynamics how the world changes after an action
r(s_t, a_t) reward the scalar score for what just happened
pi(a | s) policy the agent's behavior rule
Trajectory: tau = (s_1, a_1, s_2, a_2, ..., s_T, a_T) Trajectory distribution: p_theta(tau) = p(s_1) prod_t pi_theta(a_t | s_t) p(s_{t+1} | s_t, a_t) RL objective: J(theta) = E_{tau ~ p_theta(tau)}[sum_t r(s_t, a_t)]

Discounting

A discounted return is G_t = r_t + gamma r_{t+1} + gamma^2 r_{t+2} + .... Smaller gamma values make delayed rewards matter less. Larger values make the agent care more about long-term outcomes.

Practice trap. With a sparse terminal reward of +1 for success, increasing gamma does not favor speed. It makes slow success less punished. To favor quick success, decrease gamma or add an explicit time penalty.

Dense Rewards And Shaping

Dense rewards can make learning much easier because they provide feedback before the terminal event. But a dense progress reward is not automatically policy-preserving. It can make the agent chase local progress, avoid necessary detours, or prefer risky shortcuts. The special case that is known to preserve optimal policies is potential-based shaping, which adds a difference of potentials rather than an arbitrary progress bonus.

Potential-based shaping idea: r'(s,a,s') = r(s,a,s') + gamma Phi(s') - Phi(s) Intuition: the shaping terms telescope over time, so they guide learning without changing which full trajectories are optimal.

Reward Transformations

Change Always preserves optimal policy? Reason
Multiply every reward by positive constant c > 0 Yes All policy values and Q-values are scaled by c; action ordering is unchanged.
Add constant forever in infinite-horizon discounted continuing task Yes Every policy receives the same extra c / (1 - gamma).
Add constant per step in episodic task with variable length No Positive constant becomes a survival bonus; negative constant becomes a time penalty.
Change gamma No It changes the tradeoff between sooner and later rewards.
Apply nonlinear increasing transform to each reward No RL optimizes sums of rewards, not individual reward rankings.
Check: two trajectories have rewards [1, 1, 4] and [2, 2, 2]. Before and after squaring, which is better?
Answer. Before squaring, both total 6. After squaring, the totals are 1 + 1 + 16 = 18 and 4 + 4 + 4 = 12. A nonlinear increasing transform changed the trajectory ordering.

Imitation Learning

Behavior Cloning

Behavior cloning is supervised learning on expert demonstrations. Dataset D_expert contains state-action pairs from an expert. The policy learns pi_theta(a | s).

Behavior cloning objective: min_theta E_{(s,a) ~ D_expert}[-log pi_theta(a | s)]

If actions are continuous and the policy is Gaussian, this often looks like squared-error regression. If actions are discrete, it looks like cross-entropy classification.

Why Simple Regression Can Fail

Expert actions can be multi-modal. In driving, the expert might go left or right around an obstacle. Squared-error regression may average those modes and produce "go straight into the obstacle." The lecture's fix is expressive policy distributions: categorical distributions, Gaussian mixtures, autoregressive policies, diffusion policies, or flow matching.

Flow Matching For Policies

The reference sheet includes flow matching because it is one way to represent a rich action distribution. The idea is to start from noise a_{t,0} and learn a velocity field that transports it toward a real action a_t.

ODE: d a_{t,tau} / d tau = v_theta(s_t, a_{t,tau}, tau) Euler integration with n steps: a_{t,tau + 1/n} = a_{t,tau} + (1/n) v_theta(s_t, a_{t,tau}, tau) Flow matching loss: L_FM(theta) = (1/|D|) sum_{(s_t,a_t) in D} || v_theta(s_t, a_{t,tau}, tau) - (a_t - a_{t,0}) ||_2^2

Compounding Errors

In offline behavior cloning, the learned policy trains only on expert states. At test time it may make a small mistake, visit states the expert never visited, and then make bigger mistakes. This is distribution shift caused by the policy's own errors.

DAgger And Online Interventions

DAgger-style methods reduce compounding errors by collecting data where the learned policy actually goes. Roll out the policy, ask the expert to label or intervene at encountered states, aggregate the data, and retrain. This is stronger than pure offline BC, but requires online execution and expert access.

Check: why can behavior cloning fail even if validation loss on expert data is low?
Answer. Validation loss measures expert-distribution states. Deployment states come from the learned policy. A tiny action error can move the agent into unfamiliar states, where the supervised model has no reason to be accurate.

Policy Gradients

The Big Idea

Policy gradient is trial-and-error learning. Sample actions from the current policy. If a trajectory gets high reward, increase the probability of actions in that trajectory. If it gets low reward, decrease them.

Vanilla REINFORCE: grad_theta J(theta) = E_{tau ~ p_theta(tau)}[ (sum_t grad_theta log pi_theta(a_t | s_t)) (sum_t r_t) ]

Why The Dynamics Disappear

The trajectory probability includes environment terms:

p_theta(tau) = p(s_1) prod_t pi_theta(a_t | s_t) p(s_{t+1} | s_t, a_t)

But p(s_1) and p(s_{t+1} | s_t, a_t) do not depend on theta. Therefore, their gradients are zero. The policy-gradient estimator only needs grad log pi_theta(a_t | s_t).

Causality: Reward-To-Go

Action a_t cannot affect rewards before time t. So we replace total return with future return:

With causality: grad_theta J(theta) = E[ sum_t grad_theta log pi_theta(a_t | s_t) (sum_{t' = t}^T r_{t'} - b) ]

This keeps the expected gradient correct and usually reduces variance.

Baselines

A baseline subtracts predictable reward so the gradient focuses on better-than-expected actions. The key rule from the practice questions:

A baseline is unbiased if, after conditioning on s_t, it does not depend on the sampled action a_t. It can be wrong or poorly trained and still be unbiased. It may just fail to reduce variance.
Why state-only baseline is unbiased: E_{a ~ pi(.|s)}[grad log pi(a|s) b(s)] = b(s) sum_a grad pi(a|s) = b(s) grad 1 = 0
Check: which baselines are valid: V(s_t), E_a[Q(s_t,a)], Q(s_t,a_t), or a bad neural net c(s_t)?
Answer. Valid: V(s_t), E_a[Q(s_t,a)], and c(s_t). Invalid in general: Q(s_t,a_t), because it depends on the sampled action.

Bias And Variance

Bias means the estimator is wrong on average. Variance means it changes a lot across samples. In RL, a high-variance unbiased gradient may still be hard to learn from.

Trick Effect Why
Reward-to-go Lower variance, no bias Removes rewards that the action could not have caused.
State baseline Usually lower variance, no bias Subtracts state-dependent expected return.
Action-dependent baseline Can bias Its expected score-function term is not generally zero.

On-Policy Versus Off-Policy

Vanilla policy gradient is on-policy: rollouts must come from the current policy. If data comes from an older policy, importance sampling can correct it:

Importance ratio: rho_t(theta') = pi_{theta'}(a_t | s_t) / pi_theta(a_t | s_t)

But large ratios cause high variance and unstable updates. That motivates KL constraints and PPO.

Actor-Critic And PPO

Value, Q, And Advantage

V^pi(s) = expected future reward from state s, then follow pi Q^pi(s,a) = expected future reward from state s, take action a, then follow pi A^pi(s,a) = Q^pi(s,a) - V^pi(s)

The actor is the policy. The critic is the value or Q estimator. The actor asks, "What should I do?" The critic answers, "Was that action better or worse than expected?"

Actor-critic policy gradient: grad_theta J(theta) approx (1/N) sum_i sum_t grad_theta log pi_theta(a_{i,t} | s_{i,t}) * A_hat(s_{i,t}, a_{i,t})

Estimating Values

Estimator Formula idea Bias / variance
Monte Carlo Use actual future return from rollout Low bias, high variance, waits until trajectory data exists
1-step TD r_t + gamma V_hat(s_{t+1}) More bias from bootstrap, lower variance
n-step Use n real rewards, then bootstrap Middle ground
1-step advantage estimate: A_hat(s_t,a_t) approx r_t + gamma V_hat(s_{t+1}) - V_hat(s_t)

GAE

Generalized advantage estimation averages advantage estimates with different horizons. Short horizons have lower variance but more bootstrap bias. Long horizons have lower bootstrap bias but more rollout variance. GAE blends them with weights controlled by lambda.

n-step advantage: A_hat_n(s_t,a_t) = sum_{t'=t}^{t+n} gamma^{t'-t} r_{t'} - V_hat(s_t) + gamma^n V_hat(s_{t+n}) GAE: A_hat_GAE = sum_n w_n A_hat_n, where w_n proportional to lambda^{n-1}

PPO

PPO lets you take multiple gradient steps on a batch while preventing the new policy from moving too far. It does this by clipping the importance ratio.

ratio: r_t(theta') = pi_{theta'}(a_t | s_t) / pi_theta(a_t | s_t) PPO clipped surrogate: sum_{i,t} min( r_t(theta') A_hat_t, clip(r_t(theta'), 1 - eps, 1 + eps) A_hat_t )

If an update would make the policy probability change too much in a way that improves the surrogate, clipping removes the extra incentive.

Check: if advantage is positive, what does PPO clipping prevent?
Answer. It prevents increasing the probability of that action by more than about 1 + eps within the surrogate objective. The policy can still change, but the objective stops rewarding excessive probability increases on that sample.

Off-Policy Actor-Critic Beyond PPO

PPO is only mildly off-policy: it reuses a recent batch for several gradient steps and controls drift. Fully off-policy actor-critic methods, such as SAC-style methods, keep a replay buffer and learn a critic from older data. The policy can then be improved using the critic instead of only using fresh reward-to-go estimates.

Ingredient What changes from vanilla policy gradient? Why it helps
Replay buffer Train on transitions from many past policies Better sample efficiency, especially when real-world rollouts are expensive
Q critic Estimate Q^pi(s,a) with TD-style bootstrapping Gives a reusable estimate of action quality
Policy update through Q Increase probability of actions the critic rates highly Avoids waiting for full rollout returns every update
Importance weights or constraints Correct or limit mismatch between data policy and current policy Controls bias and instability from stale data

The tradeoff is standard for this course: more off-policy reuse improves data efficiency but creates more distribution mismatch. PPO is stable and easy to use; replay-buffer actor-critic methods are more data-efficient but more sensitive to critic quality and implementation details.

Q-Learning And DQN

From Policy Evaluation To Optimal Control

Actor-critic estimates Q^pi for the current policy. Q-learning tries to estimate Q*, the Q-value of the optimal policy, directly.

Bellman equation for a fixed policy: Q^pi(s,a) = r(s,a) + gamma E_{s', a' ~ pi(.|s')}[Q^pi(s',a')] Bellman optimality equation: Q*(s,a) = r(s,a) + gamma E_{s'}[max_{a'} Q*(s',a')]

Q-Learning Target And Loss

1-step Q-learning target: y_i = r_i + gamma max_{a'} Q_hat_phi(s'_i, a') Q-learning loss: L(phi) = (1/N) sum_i || Q_hat_phi(s_i,a_i) - y_i ||^2

Once Q_hat is good, the policy is greedy: choose argmax_a Q_hat(s,a).

Why DQN Needs Stabilizers

Stabilizer What it does Practice-problem wording
Replay buffer Stores past transitions and samples random mini-batches Breaks temporal correlation and reuses data for efficiency
Target network Computes target with a frozen or slowly updated copy of Q Stops the network from chasing a target that moves every gradient step
Exploration Epsilon-greedy or Boltzmann action sampling Gets coverage of more state-action pairs
Practice trap. The online and target networks are not kept identical throughout training. If they were identical, the bootstrapped label would move with every update.

Overestimation And Double DQN

The max in Q-learning tends to select actions whose values are overestimated by noise. Double DQN separates action selection from action evaluation: use one network to pick the best action and another to evaluate it.

n-Step Q-Learning

n-step Q target: y_{i,t}^{(n)} = sum_{t'=t}^{t+n-1} gamma^{t'-t} r_{i,t'} + gamma^n max_a Q_hat(s_{i,t+n}, a)

Larger n uses more observed rewards before bootstrapping. This reduces dependence on inaccurate early Q-values, but increases variance and can introduce off-policy bias.

Check: why is 1-step Q-learning naturally compatible with off-policy replay?
Answer. After observing (s_t, a_t, r_t, s_{t+1}), the next action in the target is chosen by max_a Q(s_{t+1},a), not by the action actually taken next in the replay trajectory. For n > 1, intermediate rewards depend on behavior-policy actions in the buffer.

Offline RL

The Problem

Offline RL trains only on a fixed dataset D collected by a behavior policy pi_beta. You cannot collect new rollouts. The goal is to learn a better policy pi_theta.

Why Not Just Use Off-Policy Q-Learning?

A learned policy may choose actions not represented in the dataset. The Q-function may be wildly wrong there. Maximizing over those unseen actions creates overestimation and bad policies.

Offline RL has one recurring principle: improve the policy while staying close to actions supported by the data.

Filtered Behavior Cloning

Keep the top-return trajectories and behavior-clone them. This is simple but crude: it ignores the fact that some good actions may appear inside medium-return trajectories.

Check: returns are {3, 9, 1, 7, 5}. If you keep the top 2 trajectories, what is their mean return?
Answer. Top two returns are 9 and 7, so the mean is (9 + 7) / 2 = 8.

AWR: Advantage-Weighted Regression

AWR turns imitation learning into weighted imitation: copy dataset actions more strongly when their estimated advantage is high.

AWR objective: max_theta E_{(s,a) ~ D}[ log pi_theta(a | s) exp((1/alpha) A_hat(s,a)) ]

Good: policy is trained only on dataset actions, so it avoids querying OOD actions. Weakness: basic AWR uses Monte Carlo returns, which can be high variance, and its advantages are tied to the behavior policy pi_beta.

IQL: Implicit Q-Learning

IQL addresses AWR's weaknesses without explicitly maximizing over out-of-dataset actions. It learns V with an asymmetric expectile loss, learns Q by bootstrapping to V, then extracts a policy with AWR-style weighted behavior cloning.

Expectile loss for residual x = V(s) - Q(s,a): ell_lambda^2(x) = (1 - lambda) x^2 if x < 0 lambda x^2 if x >= 0 IQL value fit: V_hat(s) <- argmin_V E_{(s,a)~D}[ell_lambda^2(V(s) - Q_hat(s,a))] IQL Q update: Q_hat(s,a) <- argmin_Q E[(Q(s,a) - (r + gamma V_hat(s')))^2] Policy extraction: pi_hat <- argmax_pi E_{(s,a)~D}[ log pi(a|s) exp((1/alpha)(Q_hat(s,a) - V_hat(s))) ]
Upper expectile convention in the reference sheet. The sheet says upper expectile uses lambda < 1/2 because the residual is defined as x = V(s) - Q(s,a). With this sign convention, under-placing V relative to high Q values is penalized more.
Check: what two AWR weaknesses does IQL address?
Answer. Basic AWR's Monte Carlo returns can be high-variance, and its advantages are tied to the behavior policy pi_beta. IQL uses TD-style learning and estimates values for a better policy within dataset support.

Reward Learning And RLHF

Where Do Rewards Come From?

Many real tasks do not come with a clean reward function. Reward learning tries to infer a reward from goal examples, demonstrations, classifier labels, or human preferences.

Goal Classifiers

If you can collect examples of successful and unsuccessful states, train a classifier C(s) that predicts success. Use its output as a reward signal. This gives dense feedback if the classifier can generalize.

Preference Learning

Humans often find pairwise comparisons easier than calibrated scalar ratings. Instead of asking "what reward score is this trajectory?", ask "which trajectory is better?"

Trajectory reward: r_theta(tau) = sum_{(s,a) in tau} r_theta(s,a) Bradley-Terry preference model: P_theta(tau_w preferred to tau_l) = sigmoid(r_theta(tau_w) - r_theta(tau_l)) Reward model objective: max_theta E_{(tau_w,tau_l)}[ log sigmoid(r_theta(tau_w) - r_theta(tau_l)) ]

RLHF Pipeline

  1. Start from a pretrained or instruction-tuned language model.
  2. Generate multiple responses for prompts.
  3. Collect human pairwise preferences.
  4. Train a reward model to predict preference labels.
  5. Use RL, often PPO-style, to optimize the language model against the learned reward, usually with a KL penalty to stay close to the original model.

DPO

Direct Preference Optimization avoids training an explicit reward model and running an RL loop. It turns preference data directly into a classification-style objective for the policy. The lecture frames it as simpler and effective, but less able to leverage online data.

Failure Modes

  • Humans can be inconsistent.
  • Reward models can be wrong outside the comparison data.
  • Optimizing a learned reward too hard can exploit reward-model errors.
  • KL penalties and reference models limit drift but do not make the reward perfect.
Check: why prefer pairwise preferences over scalar reward labels?
Answer. Pairwise comparisons are usually easier and more consistent for humans than assigning calibrated scalar scores. The reason is human judgment quality, not a generic theorem about lower variance.

RL For LLM Reasoning

The reasoning lecture connects RL to test-time compute. A model can improve answers not only by having better weights, but by spending more inference compute on planning, search, or verification.

Chain Of Thought And Majority Vote

Sample multiple reasoning traces, score or verify them, and choose an answer by consensus or by a verifier. This is test-time search over candidate solutions.

GRPO-Style Training

Group Relative Policy Optimization compares multiple sampled completions for the same prompt and uses relative scores within the group as the learning signal. The high-level idea is advantage estimation without a separate value model.

Exam relevance is likely conceptual: RL is used to teach policies to produce better reasoning traces when rewards are verifiable, and test-time compute changes how we evaluate capability.

Model-Based RL

Core Idea

Model-based RL learns or uses a model of the world, usually p_theta(s_{t+1} | s_t, a_t). Sometimes it also learns a reward model r_theta(s_t,a_t).

Dynamics model: p_theta(s_{t+1} | s_t, a_t) Reward model: r_theta(s_t, a_t)

Use 1: Generate Synthetic Data

Dyna-style model-based policy optimization alternates between collecting real data, fitting the model, rolling out the model to create synthetic transitions, and training a model-free RL algorithm on real plus synthetic data.

Model rollouts should often be short and start from real states in the replay buffer. Long model rollouts compound model error.

Use 2: Planning At Test Time

The planner itself can be the policy. At state s_t, sample candidate action sequences, roll them out in the learned model, score them, execute the first action of the best sequence, then replan next step.

Model rollout setup: sample N action sequences a^i_{t:t+H} roll them out with p_theta to get s_hat^i_{t+1:t+H+1} Planning V1: i* = argmax_i sum_{t'=t}^{t+H} r(s_hat^i_{t'}, a^i_{t'}) Planning V2: i* = argmax_i [ sum_{t'=t}^{t+H} r(s_hat^i_{t'}, a^i_{t'}) + V_hat(s_hat^i_{t+H+1}) ]

Why Greedy One-Step Reward Is Not Enough

Choosing argmax_a r_theta(s_t,a) ignores future consequences. The pancake robot from the practice problem might get immediate syrup reward while ruining the final pattern. Planning uses multi-step predicted return.

Why Exact Dynamic Programming Is Usually Intractable

If states are images or continuous robot configurations, you cannot enumerate all possible (s,a) pairs. A discrete action space does not make the state space small.

Check: for a high-dimensional kitchen robot with learned transition and reward models, which are plausible: one-step greedy, sampling-based planning, exact DP over all images, synthetic rollouts?
Answer. Plausible: sampling-based planning with replanning, and synthetic rollouts for model-free training. Not generally plausible: one-step greedy reward, exact DP over all image states, or exact backward search from the final image.

Multi-Task And Goal-Conditioned RL

Multi-Task RL

Instead of one MDP, we have a task distribution. Tasks may differ in rewards, dynamics, initial states, action spaces, or goals. The key practical idea is to condition the policy on the task.

Task-conditioned policy: pi_theta(a | s, z) where z can be a task ID, goal state, instruction, desired object, user ID, etc.

Why Share Across Tasks?

  • Shared representations can reduce sample complexity.
  • Data from one task may teach dynamics or skills useful for another.
  • But imbalanced data can cause common tasks to dominate training, so stratified sampling can help.

Goal-Conditioned RL

Goal-conditioned RL is a special case where z is a goal, often a goal state s_g. The policy learns pi(a | s, s_g).

Hindsight Relabeling / HER

If an episode failed at its intended goal but reached some other state, relabel that reached state as the goal and train as if the episode succeeded for that alternate goal.

Hindsight relabeling needs compatible dynamics across goals or tasks and is naturally paired with off-policy algorithms using a replay buffer.
Check: when is hindsight relabeling valid?
Answer. When the same trajectory can be reinterpreted under a different goal or task without changing the dynamics. You can relabel rewards and the task identifier, store the relabeled transition in replay, and train off-policy.

Practice Problem Patterns

Pattern 1: Reward Transformations

Ask: does this transform add the same constant or positive scale to every policy's return? If yes, policy ordering is preserved. If it changes time preference, episode length incentive, or sums nonlinearly, policy can change.

Pattern 2: Baseline Validity

Condition on s_t. If the baseline still depends on the sampled action a_t, it can bias the gradient. If it depends only on state, prompt prefix, time, or a frozen model reading the prefix, it is unbiased even if inaccurate.

Pattern 3: DQN Stability

Replay buffer gives decorrelation and data reuse. Target network gives a slower moving label. Both are about making bootstrapped neural-network regression less unstable.

Pattern 4: n-Step Returns

More observed rewards means less bootstrap bias and often faster early learning. But in replay data, those intermediate rewards came from behavior-policy actions, so large n can introduce off-policy bias.

Pattern 5: Offline RL And IQL

If a method maximizes over arbitrary actions, ask whether those actions are in the offline data. If not, beware OOD overestimation. AWR and IQL keep policy training on dataset actions.

Pattern 6: RLHF Preferences

Pairwise preferences are used because humans compare options more reliably than they assign absolute scalar scores. Bradley-Terry converts reward differences into preference probabilities.

Pattern 7: Model-Based Planning

A learned reward model is not enough for good decisions. Long-term planning requires rolling out dynamics. Exact DP is impossible in high-dimensional continuous/image states, but sample-based planning or synthetic rollouts can be tractable.

Rapid Questions

1. A baseline is trained badly but depends only on s_t. Biased?
Answer. No. It may not reduce variance, but it is action-independent.
2. You square all positive one-step rewards. Does the optimal policy always stay the same?
Answer. No. Sums of transformed rewards can reorder trajectories.
3. Why does a target network help DQN?
Answer. It makes bootstrapped regression labels change slowly instead of changing every time the online network updates.
4. What is the offline RL danger of max_a Q(s,a)?
Answer. The max can select out-of-dataset actions whose Q-values are overestimated because the dataset gives no corrective evidence.
5. Why add a terminal value to model-based planning?
Answer. It lets short-horizon planning account for rewards beyond the rollout horizon, reducing myopia in long-horizon tasks.
